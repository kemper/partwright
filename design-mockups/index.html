Rethinking the layout & flow

Four throwaway prototypes exploring how to make a feature-rich, browser-based, AI-driven CAD tool easier to navigate. Same fake scene in every mock so you can compare layouts apples-to-apples. Static HTML, no build step — open any file in a browser.

What the current app does well — and where it strains

The mental model is sound: a landing portal → one editor workspace (code + viewport) → AI drawer, with sessions/versions as the core unit. But years of organic feature growth have produced three competing control zones and a lot of unlabeled iconography. The recurring theme: capability is high, but discoverability lags.

01

Icon soup

~14 toolbar controls + a crowded viewport-corner overlay, many glyph-only (⚙ ⚠ ? 🎯 ✦, △ ▦ 📏 🔓). Meaning lives only in tooltips. The ruler 📏 means two different things in the same bar.

02

Three overlapping control zones

Top toolbar, session bar, and the viewport overlay all hold "global-ish" actions. View toggles, measure, paint, annotate, simplify and cross-section all pile into one wrapping corner that collides with the gizmo and clip slider.

03

Tab overload & redundancy

Seven right-pane tabs (Interactive / Gallery / Versions / Images / Diff / Notes / Data). Gallery vs Versions overlap heavily; Data is a debug surface sitting beside everyday tabs; Diff hides the editor.

04

Split project model

"Recent sessions" lives on the landing page, but the full session manager is a modal reachable only from inside the editor. No single "my projects" home; versions are managed across four different surfaces.

05

AI: central pitch, peripheral UI

The product sells "built for AI agents," yet in-app AI is one ✦ toggle opening a very dense drawer (11-control toggle strip, no provider switcher in the header) that's disconnected from the tab system.

06

Surprising mode coupling

Painting silently locks the editor and forks a version; overlay tools are mutually exclusive; ⌘Z routes to whichever tool is active. Powerful, but the state changes aren't anticipated.

Four directions to compare

Each prototype attacks discoverability with a different organizing principle. They aren't mutually exclusive — the best answer is likely a blend (see the recommendation).

Concept A

Classic menu bar + labeled toolbar

Give every action a discoverable, categorized home — File / Edit / View / Insert / Measure / AI / Help — plus a contextual toolbar where every tool is icon and word.
+Most familiar; exhaustive & self-documenting; shortcuts shown inline; nothing is a mystery glyph.
+Scales to dozens of features without adding screen clutter.
–Feels like a desktop app; menus hide depth behind clicks; weakest on mobile/touch.

Concept B

Left activity rail + clean toolbars

One labeled vertical navigation spine replaces the seven cryptic tabs; a single labeled toolbar pinned to the viewport replaces the crowded corner. Merges Gallery + Versions.
+Modern (VS Code / Figma) feel; a calm, labeled home for navigation; collapses to icons or a mobile bottom-bar gracefully.
+AI sits naturally as a first-class destination in the spine.
–Costs ~56–210px of permanent width; biggest structural change to today's code.
Open prototype →
Concept C

Command-first / minimal chrome

Keep the surface extremely calm (editor + viewport + AI). Make a powerful, searchable, categorized ⌘K palette the primary way to find and do everything.
+Maximum focus on the work; fastest for power users; everything labeled & searchable in one place.
+Builds on the palette the app already has — smallest lift to start.
–Discoverability still depends on people finding the palette; weak for first-timers who don't know what to search.
Open prototype →
Concept D

Task workspaces / modes

Split into a few named workspaces — Model · Decorate · Review · Inspect. Each shows only its relevant tools, so you never see everything at once and the paint→lock becomes an expected mode switch.
+Dramatically reduces simultaneous complexity; turns surprising state changes into intentional ones; great for onboarding.
+Each mode can be tuned for its job (and its device).
–Adds switching cost; risks hiding a tool a user wants "right now"; needs clear signposting.

My recommendation — a blend, phased

No single concept wins outright; each solves a different slice. A pragmatic path that keeps the browser-only, AI-forward identity:

  • Foundation — Concept B's shell. A labeled left rail is the single biggest discoverability win and the cleanest way to retire the seven-tab strip and the crowded corner overlay. Merge Gallery+Versions; give AI a real home in the spine.
  • Always-on — Concept C's palette. Invest in one great categorized ⌘K as the universal accelerator and the home for rarely-used/destructive actions (reset data, diagnostics, tour) so they stop hiding. Cheap, complements any shell.
  • Labels everywhere — Concept A's discipline. Adopt the "icon + word" rule and inline shortcut hints for every persistent control. You don't need the full menu bar to steal its self-documenting habit.
  • For heavy modes — Concept D's idea, scoped. Don't mode the whole app, but make Decorate (paint + annotate on a locked snapshot) an explicit workspace so the silent paint-lock becomes a deliberate, understood switch.

These are disposable visual studies — fake content, no real behavior, no shared code with the app. Built to spark a direction conversation, not to ship. Open a/b/c/d-*.html directly in any browser; each is fully self-contained (no network, no dependencies).
