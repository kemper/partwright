## Key decisions

**Reused `api.joints.ballSocket` instead of building captive print-in-place
joints from scratch.** The codebase already has a tested ball-and-socket
primitive (the `joints` namespace, soozafone-spirit but mature) with
friction-finger retention. Each Dummy 13 part is printed separately and
snap-assembled — the print-in-place captive joint is a much harder tolerance
problem (and a one-time setting that locks every joint to the same FDM
calibration). The friction socket holds pose without per-printer tuning.

**`api.dummy13.*` as a layered namespace, not duplicated boilerplate in
each example.** Followed the same pattern as `joints.ts`/`fasteners.ts` —
`createDummy13Namespace(mod, { joints })`. Standard sizes/proportions live
in one place (`DUMMY13_DEFAULTS` + `DUMMY13_PROPORTIONS`), keyed off a
single `height` knob. Pure helpers (`resolveSpec`, `segmentLengths`,
`cupGeometry`, `ballEndHeight`) get unit tests; the geometry builders are
exercised by the catalog bake.

**13 parts split into 8 catalog entries.** A symmetric pair (e.g. left +
right upper arms) is one printable JS source that renders both side by side
— so the catalog count is 8 (head, torso, hips, upper_arms, forearms,
hands, thighs, shins) for 1+1+1+2+2+2+2+2 = 13 physical parts. Plus a
9th `dummy13_full_skeleton` entry that's a visualization-only assembly.

**Skipped photo→Dummy 13 (Phase 3 from my earlier feasibility note).** That
path needs joint-envelope collision checking that doesn't exist in the
codebase, plus a way to slice the existing figure pipeline output at joint
planes. Tractable but a real new pipeline — explicitly out of scope for
this PR.

**`fullSkeleton` T-pose has a known cosmetic quirk.** The upper-arm body
overlaps the torso visually in the assembled view because of how the limb's
local frame rotates: the cup mouth and body extension are along the same
local axis, so a single rotation can't put the cup at the shoulder AND
extend the body outward without overlap. The geometry is correct for each
individual standalone part (which is what users print). The fullSkeleton is
a visualization aid, not a printable thing — the caption says so. A clean
fix would restructure `buildLimb` so cup mouth and body extend in opposite
local directions, but that complicates the print-orientation of each
standalone part (the cup base no longer sits flat on z=0). Punted; flagged
in the assistant entry instead of half-fixing.

**Defaults sized for a ~135mm figure, FDM 0.4mm nozzle.** Major joints
(shoulder/hip) at 8–9mm ball diameter; limb joints (elbow/knee/wrist) at
4.5–6mm. clearance=0.18mm, openingRatio=0.85 — friction grip without
forcing-fracture insertion. User can override any of these per-part via
the `height` param; finer tolerance knobs are in `DUMMY13_DEFAULTS` for
future param exposure if tuning needs it.

**Print-tested flag stays false.** Per the catalog convention, only flip
`printTested: true` after a verified physical print. The user is going to
print these and report back — that's the validation pass.
