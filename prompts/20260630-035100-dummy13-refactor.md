## Key decisions

**Got the official STLs via Printables' public GraphQL API.** The `print()`
query exposes `stls { id name folder fileSize }`, and the `getDownloadLink`
mutation returns a signed URL. No auth required. Downloaded all 21 frame +
hand STLs to scratchpad.

**License: CC-BY 4.0 — verified via GraphQL.** Most permissive CC short of CC0:
remix and commercial use both allowed with attribution. Path is unblocked for
a parametric reimplementation.

**The universal joint spec: every socket cavity is exactly 6.0 mm.** Reverse-
engineered by least-squares sphere fits across `frame_hips`, `frame_chest`,
`frame_abdomen`, `frame_waist`, `frame_neck`, `frame_clavicle`, `frame_forearm`,
`frame_shin` — all 6.0 mm, RMSE = 0. This is soozafone's design genius — the
universal spec is what lets the `hip_and_shoulder_4x` (one piece, four joint
positions) and `knee_and_elbow_4x` (one piece, four positions) shared parts
work. Made it the central pinned invariant: `DUMMY13_SPEC.socketCavityD = 6.0`,
with a unit test that fails if anyone changes it.

**Ball diameter: 5.7 mm default vs measured 5.0 mm stock.** Sphere fit on a
hip/shoulder bridge ball gave 5.0 mm — which produces the well-documented
"loose Dummy 13 joints" complaint (0.5 mm radial clearance per side, too
sloppy to hold pose). Ship at 5.7 mm by default for tighter friction-finger
hold; exposed `ballD` as the first param on every catalog entry so anyone who
wants to print stock-spec parts that interop with soozafone's STLs can set
`ballD: 5.0`.

**Architecture rewrite: 28 parts, not 13.** My first cut had 13 parts with
balls on the body and sockets on the limbs. The real Dummy 13 has the
opposite distribution:
  - 6 body segments (head/chest/abdomen/waist/hips/feet) — ALL sockets, no balls
  - bridge pieces (neck/clavicle ×2/hip-shoulder ×4/knee-elbow ×4/ankle ×2)
    are the ONLY balls — every joint ball lives on a small interchangeable
    bridge
  - 8 limb segments (upper arm/forearm/thigh/shin ×2 each) — sockets on both
    ends
That's 28 printable parts. The "13" in Dummy 13 is a name (from Lucky 13
Toys), not a count; at 100% scale the figure is ~13 cm tall — possibly the
real origin of the name.

**Dimensions all measured, not guessed.** Hip width 16 mm (sockets at x = ±8
on the hips bar). Shoulder width 12 mm. Each body segment height taken from
the STL bbox Y axis. Total skeleton ~165 mm at 100% scale matches the
community lore that the Etsy "11-inch" sellers run at ~215% scale.

**Catalog architecture: 17 entries for 28 parts.** Pairs and 4× sets share
one source file via `dummy13.plateOf(part, count, gap)` — same pattern as
the first cut, but now with the proper architecture (separate
`dummy13_clavicles` / `dummy13_ankles` / `dummy13_neck` / `dummy13_hip_shoulder_bridges`
/ `dummy13_knee_elbow_bridges` / `dummy13_feet` entries the first cut was
missing). `dummy13_full_skeleton` stays as the visualization-only entry.

**Visual fidelity is approximate; spec fidelity is exact.** My
parametric parts use simple capsule + flat-bar primitives, not soozafone's
silhouettes. The body segments are taller than official because I stack
`cup → bar → cup` whereas the official mills cavities into a flat bar (no
extra height per socket). Joint interop is exact (6 mm socket, configurable
ball Ø); aesthetics are simple. Per user's "parametric refactor only"
choice — they explicitly opted against embedding the official STLs.

**Attribution everywhere.** Every catalog entry description ends with
"Compatible with soozafone's Dummy 13 v1.0
(https://www.printables.com/model/981111, CC-BY 4.0)". The module header
calls out the CC-BY attribution requirement explicitly.

**What I deliberately deferred.** The hands/feet attach with sockets only;
to fully assemble against soozafone's parts you'd need a small wrist bridge
piece between hand-socket and forearm-socket — not yet exposed as its own
entry, but the user can build one by calling `api.dummy13.kneeElbowBridge`
(same geometry, smaller bridge). Also: my `fullSkeleton` preview has the
arms hanging straight down rather than the soozafone-default T-pose, because
of the same single-axis-rotation issue I flagged in the first cut's prompt
log. Functional but not picturesque. Cosmetic; users print individual parts.
