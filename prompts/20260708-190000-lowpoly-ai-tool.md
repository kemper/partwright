## Human

Explore adding a "low poly mode" as an AI tool, in two directions: (1) convert an
uploaded STL/model to a cool low-poly version — maybe an alternate way of doing
the existing STL→code conversion; (2) give an AI a photo and have it create a
model in a low-poly style. Today the AI keeps unioning separate primitives and it
looks nothing like traditional low-poly; it probably needs a tool that explicitly
creates low-poly the way the voxel engine makes voxels. Explore both, then build
directions 1 and 2. (Plus feedback from a prior AI attempt: wants a
`lowPoly(node, { targetTriangles/facetSize, flatShade })` that decimates to a
triangle BUDGET, not a raw edgeLength; notes painting is a separate weaker gap
and `api.paint.pattern` should be the primary coat workflow.)

## Assistant

Exploration (4 parallel subagents) found the two ideas collapse onto one
primitive — a coarse mesh + flat shading — and most of it already existed:

- A full triangle-decimation engine (`src/geometry/simplify.ts`, binary-search to
  a triangle budget) was already built but surfaced only as the UI-only
  "Simplify/Enhance" panel — no API/tool/in-code path.
- The renderer already flat-shades any mesh carrying per-triangle colours (it
  unindexes per triangle, so `computeVertexNormals` yields face normals). So the
  faceted look needed no new material — just extend that unindexed path to an
  unpainted low-poly mesh.
- The root cause of the bad photo→creature results was a *system-prompt* line
  that routed "low-poly" OUT of SDF and back into the "primitive soup" the same
  prompt calls wrong — not a missing capability.
- The STL→code reconstruct pipeline is SDF/levelSet codegen and explicitly warns
  against post-hoc simplification, so low-poly is a *separate* path, not a mode
  of it.

Decision: build ONE in-code operator that serves both ideas, plus the guidance
fix, rather than a whole new "vertex-pushing" engine (deferred — 10× the effort
for marginal gain over crystallize-a-blob).

- `api.lowPoly(shape, { targetTriangles?, facetSize?, flatShade? })` in the
  manifold-js sandbox. Targets a triangle BUDGET by default (even facet size —
  the prior AI's specific ask), returns a Manifold so it drops into a `return`.
  Core decimation is a pure, synchronous sibling of `simplify.ts`
  (`lowPolyOp.ts`) — sync because sandbox code runs synchronously; unit-tested
  against a fake manifold.
- `flatShade` (default true) is carried as a per-run, non-persisted `mesh.flatShade`
  render hint (mirrors `renderOnly` — re-derived from code on reload, no schema
  bump). Viewport gains one branch: unindex + face normals when flatShade and no
  colours.
- Guidance: new `public/ai/lowpoly.md` subdoc (build SDF → crystallize → paint
  coats with `api.paint.pattern`, budget ranges, photo flow); corrected the two
  systemPrompt lines that steered low-poly into primitive soup; registered the
  subdoc in `SUBDOC_NAMES_LIST` + `readDoc`; documented `api.lowPoly` in ai.md.

Chose the sandbox-API surface (like `api.sdf`/`api.surface`) over a
`window.partwright` method for v1 — it's a modeling primitive, not a UI action.
The one-click "low-poly-ify the CURRENT model" import path (surface modifier +
`partwright` API + AI tool + UI) and the deeper coat-paint work (presets,
facet-snapping boundaries) are deferred to follow-up issues, not crammed here.

Verified in the real app viewport: a 27,242-tri smooth SDF body →
`api.lowPoly(mesh,{targetTriangles:500})` renders as 500 hard-faceted triangles,
same silhouette. Unit tier (1785) green; lowpoly e2e green.
