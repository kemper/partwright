## What's working (preserve these)

- **The bake → `Read` PNG → iterate loop is still the MVP** — confirmed again by all 6 sessions across manifold-js, scad, replicad, and voxel engines. Stats alone miss render-level defects (solid gyroid blobs that pass `isManifold: true`, hidden gondolas, too-shallow twist grooves). Eyes-on is non-negotiable for art-directed geometry.
- **`api.label(shape,'name',{color})` + `api.labeledUnion` for manifold-js** works cleanly across booleans and exports. Every modelling wave confirmed this is the headline color path — vibrant tiles with zero paint step.
- **Isolate-the-variable beats parameter-thrashing.** One retro documented this directly: stripping back to a single diagnostic primitive (does a helical slab split a cone?) resolved in one render what 6 rounds of full-assembly tuning had missed. Documented in `mechanisms.md`.
- **PR 470 shipped the `model:preview` headless CLI** with voxel + scad + manifold-js engine support — the #1 longed-for item across the two prior retros (16/16 sub-agents, 3 consecutive retros). This removes the dominant wall-clock and token cost for non-browser modeling iteration.
- **`model-sculpt` subagent pattern** — image tokens from iterative visual loops no longer compound in the main context. Delegate render→look→adjust to the subagent; ship the final PNG via `SendUserFile` without a main-context Read.

## Top friction (ranked)

### 1. [docs / sdf] Wrong TPMS `thickness` guidance caused solid-blob results — 2 independent agents

**Root cause:** `sdf.md` documented `thickness ≈ cellSize/6 to cellSize/3` as the "sweet spot" for gyroid, schwarzP, diamond. But `thickness` is a **field threshold** in the TPMS field's natural units (~0–1.5 range), not a physical wall width. For gyroid with `cellSize=10`, the old formula gives `thickness ≈ 1.67–3.33`, which selects nearly all of space → a solid blob. The actual open-lattice range for gyroid is `thickness ≈ 0.4–0.7`. Two independent agents (lattice ball, gyroid cube) followed the documentation faithfully and got solids on every attempt, burning a full revision round each.

→ **Applied:** `public/ai/sdf.md` — corrected the parameter description (field threshold, not wall width), rewrote the "Sizing `thickness`" paragraph with empirically-verified ranges (0.4–0.7 for open lattice, ≥ 1.1 → solid blob), and added the `edgeLength`-to-`cellSize` tie rule. Also fixed the Lidinoid note (field-units range, not cellSize ratio).

### 3. [docs] Agents writing unverified geometric claims into docs — 2 retros, same failure class

**Root cause:** Entry 3 (spiral-cone mechanism) documented a false topological claim ("a helical slab does NOT split a cone") to rationalize retreating from a correct design. Entry 2 (sdf.md) is the same failure class: the `cellSize/N` guidance was presumably plausible-sounding but wrong. In both cases the claim was written before engine verification, and downstream sessions inherited the error.

→ **Applied (partial):** The model:preview section of `CLAUDE.md` already carried "Verify topological/geometric claims with `model:preview`, not from memory." The componentCount/decompose discipline paragraph I added (see #4 below) reinforces this. The specific false claim from entry 3 was already corrected by the human mid-session. The `sdf.md` fix (item 1) removes the most-impactful persisted error.

→ **Backlog:** A structural rule ("reason → test → document; never document a geometric conclusion before engine verification") would be worth adding explicitly to the Planning Files / Documentation section of `CLAUDE.md`. Left as backlog — the existing rule is close enough that adding another doesn't reduce risk more than a once-off agent will observe it; the real fix is making `model:preview` fast enough that verification is never skipped.

### 4. [docs / workflow] `componentCount` misdiagnosis burned cycles — 4 agents

**Root cause:** When `componentCount` was wrong, multiple agents responded by tuning parameters on the whole assembly, not isolating which component had broken off. Some agents also second-guessed legitimately multi-part assemblies (orrery with 26 parts, watch movement with 28) as defective. The `{maxComponents}` escape hatch on `runAndSave` wasn't surfaced.

→ **Applied:** `CLAUDE.md` — added paragraph: "When `componentCount` is wrong: decompose and inspect, don't tune blindly" with the `Manifold.decompose()` pattern. Added note that `isManifold: true` is the correctness gate, not the count, and that `{ maxComponents: N }` to `runAndSave` signals intentional multi-part assemblies.

---

### 5. [docs] `api.label({color})` unavailable in replicad — doc-vs-reality gap

**Root cause:** `colors.md`'s "Color in the code" section showed `api.label(shape, name, { color })` without any engine scope. `replicad.md` described `BREP.label(shape, name)` as "the BREP equivalent of `api.label`" without noting the color argument doesn't exist. Two catalog agents (faucet, watch with BREP sessions) expected self-coloring and got gray models.

→ **Applied:** 
- `public/ai/colors.md` — added "> **manifold-js only.** The `{ color }` option on `api.label` is manifold-js only. In a replicad session, models always render gray until `paintByLabel` after `runAndSave`."
- `public/ai/replicad.md` — added inline comment to the `BREP.label` code block: no color argument, always gray by default.

---

### 6. [docs / agents] Image-token compounding in iterative visual sessions — 1 session

**Root cause:** During a photo→figurine session, the main agent Read ~10 preview PNGs across iterations. Each stayed in context and was re-billed every subsequent turn. Nothing in CLAUDE.md flagged this pattern or pointed to the subagent delegation fix.

→ **Applied:** `CLAUDE.md` model:preview section — added: "Delegate multi-pass visual iteration to the `model-sculpt` subagent. Each preview PNG you Read in the main context stays there and compounds. For 3+ render passes, delegate to model-sculpt; it returns only text and the main agent ships the final result via `SendUserFile` without reading it."
