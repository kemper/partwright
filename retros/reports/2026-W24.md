# Retro Report — 2026-W24

**Entries reviewed:** 6 (from 6 sessions, covering ~24 sub-agents)  ·  **Span:** 2026-06-05 → 2026-06-07

## What's working (preserve these)

- **The bake → `Read` PNG → iterate loop is non-negotiable for art-directed geometry.** Confirmed again across all 6 sessions: stats that look correct (`isManifold:true`, `componentCount:1`) can hide a solid gyroid blob, a spindly arm absorbed into a block, or a spiral groove too shallow to see. Eyes-on is the only gate for visual quality.
- **`api.label(shape,'name',{color})` + `api.labeledUnion`** — self-coloring without a paint step works across the manifold-js and voxel engines and survives booleans cleanly. The pattern produced vibrant catalog tiles first-try across many agents.
- **`Manifold.extrude(cs, h, nDiv, twistDeg, scaleTop)` is the twist/spiral workhorse.** Five agents independently converged on this as the right primitive for the whole fidget family (ball, cone, top, egg, star). Once you have the cross-section, it's one call.
- **`model:preview` was the #1 productivity multiplier for the single-agent tasks (coat hook, spiral-cone mechanism).** The 4-view PNG caught orientation bugs and proportion issues that reasoning steps alone missed; the ~2 s feedback loop let geometry questions be answered empirically rather than algebraically.
- **Isolating a primitive in a tiny diagnostic beats parameter-thrashing.** The spiral-cone retro is the clearest proof: switching from "tweak and re-render the whole assembly" to "test one question in 3 lines" resolved a 7-round wrong-answer slog in one render.
- **`Manifold.decompose()` for print-in-place splits.** Confirmed as the clean primitive for coloring separate parts of a slab-cut solid — far simpler than half-space math.

## Top friction (ranked by cross-agent frequency)

### 1. [tooling] No headless preview for sub-agents — 20+ agents across W23+W24
Sub-agents authoring geometry in the browser-barred contract had zero feedback: no stats, no PNG, no component count. Every depth/proportion/color guess cost a full central bake round-trip (~30–90 s cold Chromium + WASM). The ask is explicit and unanimous across three consecutive retro cycles (20260605-143000 called it "the single highest-leverage tooling gap"; 20260605-163500 called it "no longer a nice-to-have"). **Status: implemented.** The `model:preview` CLI and `model-sculpt` subagent were added in the W23/W24 cycle. CLAUDE.md was stale (only mentioned manifold-js) — **updated in this PR** to reflect `--lang voxel|scad`, the photo CLI, and the daemon path for replicad.

### 2. [docs] `sdf.md` gyroid `thickness` guidance was actively wrong — 2 agents independently
`thickness` is a field threshold on the ~[-1.5, 1.5] TPMS field, **not** a wall width or a fraction of `cellSize`. The documented "cellSize/6 to cellSize/3" ratio produced solid blobs (`thickness ≥ 1.1 ≈ solid`); the actual open-lattice range is `0.4–0.7`. Two independent agents followed the doc and wasted a full revision round on both affected catalog models. → **Fixed in this PR** (`public/ai/sdf.md`: corrected the field-threshold explanation, added the concrete open/closed/solid ranges, noted the TPMS edgeLength budget of `cellSize/14..16` and the ~160–200k triangle ceiling for catalog entries).

### 3. [docs/discipline] `componentCount` misread for multi-part assemblies — 4+ agents
Agents building legitimately multi-component assemblies (orrery 26 parts, watch 28, airship 23, pagoda 15) saw `componentCount > 1` and second-guessed correct geometry, burning build rounds trying to fuse parts that should be separate. `isManifold:true` is the real gate for printability; component count is intentional for assemblies. Additionally, when count IS wrong, the right move is `decompose()` + inspect — not blind parameter-tuning. → **Fixed in this PR** (`public/ai/visual-verification.md`: expanded item 3; `public/ai/mechanisms.md`: added explicit "when componentCount is wrong" discipline note).

### 4. [docs] Image-heavy iteration burns main-context tokens silently — 1 agent, high cost
During multi-pass photo→figurine sculpting, 10+ preview PNGs stayed in the main context and compounded as re-billed image tokens each turn — nothing in CLAUDE.md flagged this or pointed to the cheaper pattern (`model-sculpt` subagent + `SendUserFile`). → **Added note to CLAUDE.md** in the model:preview section. `docs/agent-tooling.md` already documents the `model-sculpt` agent and cost-invariant pattern; CLAUDE.md now cross-references it.

### 5. [docs] SCAD `for`-loop wrapping `label()` silently drops all labels — 1 agent, full rebuild cost
A loop wrapping `label()` triggers SCAD's auto-naming fallback, nuking every label with only an `INFO` line the bake helper swallows. `PAINT FAILED` shows empty; the model bakes gray; the unexpected object count is the only clue. `bosl2.md`'s Known Footguns section warned about labels inside booleans but not inside loops. → **Added explicit footgun to `public/ai/bosl2.md`** ("Unroll every loop that contains a label() call").

### 6. [docs] No empirical fidget/twist depth & proportion rules — 4 agents re-derived independently
The depth rule (groove crest must exceed clip radius by ≥ 0.3·R), the flat-spiral-bloom aspect ratio (height ≈ 0.25–0.35 × base radius + high twist + `scaleTop > 1`), and the "when componentCount is wrong, decompose — don't tune blindly" discipline were each independently re-derived (and first shipped wrong) per model. → **Added "Making a twist actually look twisted" section to `public/ai/mechanisms.md`** with the empirically-derived depth and proportion rules.

### 7. [docs/discipline] Unverified geometry gets written into docs — 2 agents
The gyroid `thickness` doc bug and the spiral-cone retro's wrong rotation claim were both written into reference docs before being empirically tested. The `mechanisms.md` spiral-cone post-mortem makes the discipline explicit: "verify geometric/topological claims empirically before asserting them." → **No new doc entry added** (the mechanisms.md post-mortem note covers the discipline; writing a meta-rule about meta-rules would be circular). Noted here as a pattern to watch.

## One-offs noted (no action this week)

- **`manifold.decompose()` can report `componentCount > 1` for a face-connected voxel grid** — it counts interior pockets and edge-only touches, not just 6-neighbour face connectivity. For "is this one printable piece?" a face-connectivity BFS is more trustworthy than `componentCount` alone in the voxel case. (20260607-143000 — useful nuance for voxel agents, but too narrow for a general entry.)
- **`fillet(r)` fails hard with "radius too large" on ordinary cylinder rims** — no preflight, no clamp, must delete the feature. (20260605-013000 — replicad engine behavior; needs an engine-layer fix or a `try`/fallback helper.)
- **`voxels.color(c)` has no `toHex()` companion** — round-tripping packed color → hex for bridge corrections is manual arithmetic. (20260605-013000 — small ergonomics gap; worth adding to `voxel.ts`.)
- **`Manifold.revolve` sweep plane / rotation notes in session summaries came out wrong** — the session summary incorrectly stated only one rotation was needed for a YZ-plane arc. Rotations depend on the desired end-tangent direction; empirical verification with `model:preview` is the only safe path. (20260606 — already addressed by the "verify before writing into docs" discipline.)
