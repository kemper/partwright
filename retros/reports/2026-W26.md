## What's working (preserve these)

- **`model:preview` fast loop** — proportions, pose, SDF-probing with `evaluate()`, and `--require-labels` paint QC are the correct fast-loop tools. Agents who used them well moved in seconds not minutes.
- **`figure:smoke` + `--require-labels`** — caught buried/aliased labels before the slow xvfb bake. Works as advertised; every agent who ran it caught defects that looked clean in the stat block alone.
- **Prototype-before-coding for aesthetic decisions** — the "2–3 variations, user picks direction" pattern consistently converged faster than iterating one interpretation. Multiple sessions praised it explicitly.
- **Single-writer discipline for concurrent bakes** — orchestrator owns all git/bake, subagents author only. Zero working-tree races when the pattern was followed.
- **SDF-probing with `evaluate()`** — walking the field to measure actual surface positions instead of trusting analytic bounds prevented clearance/clip-plane bugs.
- **Verify-before-act on reported defects** — sessions that re-rendered the reported case before fixing confirmed: half of "still broken" reports were fixed code or by-design behavior. Saved multiple multi-agent fix waves.
- **Parallel worktrees for distinct-function edits** — four agents each owning non-overlapping functions in one large file auto-merged with zero conflicts. The constraint "non-overlapping" is key.

## Top friction (ranked by cross-agent frequency)

### 1. SSR vs browser `componentCount` — agents re-discover the oracle gap every session
**~12 sessions** · Cost: very high (each re-discovery burns 20–60 min of fix-wave time)

The headless Node mesher (`model:preview`) under-reports `componentCount` for near-threshold welds and cannot validate paint-label resolution. Agents repeatedly reported `componentCount:1, isManifold:true` with confidence from `model:preview`, then discovered the browser bake split the same figure into 5 components (marginal welds) and painted 0 triangles for eyes. The fix wave from this single misunderstanding cost more total agent-hours than any other single issue in the inbox.

**Root cause:** The callout existed in CLAUDE.md but was a single sentence in a subordinate note. Not prominent enough to interrupt confident reporting.

**Action taken:** Strengthened the `figure:smoke` callout to call out explicitly that "**The colored catalog bake is the only correctness oracle for both `componentCount` and paint-label resolution**." Also added the hair-texture edgeLength trap as a separate named warning.

---

### 2. `smoothUnion` wipes paint labels — silent defect that costs a full re-bake to discover
**~5 sessions** · Cost: high (re-bake + defect triage)

A `smoothUnion` over a fully-labelled figure node silently erases all label attribution. The SDF blends correctly but can't carry label regions. A prop attached via `smoothUnion` to a labelled figure wipes every color from the figure. Fix is simple (hard-union the prop instead), but the trap is silent — no error, no warning, just a fully gray baked figure.

**Action taken:** Added a named callout in the `smoothUnion` section: "A `smoothUnion` over a fully-labelled figure node wipes ALL paint labels."

---

### 3. Open issues after merged fixes — the "still broken" read that triggers re-investigation
**~5 sessions** · Cost: medium (20–30 min re-triage per session)

Multiple sessions began by re-investigating "open bugs" that were actually fully fixed — the merged fix PRs had left their source issues open. The existing guidance correctly says "a merged PR is not automatically a completed issue," but the inverse was missing: once criteria ARE met, close the issue. A completed fix that leaves its issue open is indistinguishable from an unfixed bug.

**Action taken:** Added the inverse rule to Issue hygiene point 2: "The reverse is equally important: once every criterion IS met, close the issue."

---

### 4. Open-ended triage agent prompts — high false-positive rate wastes expensive parallel runs
**~4 sessions** · Cost: high (6 agents × 20–30 min = 2–3 hours per occurrence)

"Find all defects in these 43 figures" prompts produced flagging of normal anatomy, expected poses, and already-fixed code at high rates. One session: six parallel triage agents ran ~20–30 min each and produced flagging of "wrong pose" on grand_jete/lotus_yogi (by-design grounding), "superhero chest defect" (that was the chest emblem), and "muscled back anomaly" (normal anatomy). Acting on raw reports would have reworked correct code. The root fix was to verify agent reports by rendering the flagged case before acting.

**Action taken:** Added a new agent-discipline bullet: "Brief triage agents on confirmed defects, not open-ended sweeps."

---

### 5. Re-discovering existing tools / capabilities before implementing them
**~3 sessions** · Cost: medium (30–90 min of design + partial implementation)

The warm-browser daemon (`bin/partwright.mjs daemon start`), `compare` command, and colored `model:preview` default were each re-requested by agents who hadn't read the relevant CLAUDE.md section. One session got to partial implementation before discovering the daemon existed.

**Action taken:** Added agent-discipline bullet: "Check whether a capability already exists before building it."

---

### 6. Engine idiom copies in examples not caught when fixing the shared helper
**~3 sessions** · Cost: low-medium (1–2 figures re-broken per release)

When a fix lands in `sdfFigure.ts` / a shared helper, figures that hand-rolled the same geometry inline keep the old (now-wrong) idiom. The engine fix is invisible to them. Example: the strongman's hand-rolled areola coin used the pre-fix plug idiom; it survived two rounds of engine fixes.

**Action taken:** Appended to the "Duplicated Logic" section: "When fixing a pattern in a shared engine file, grep `examples/` for copies of the OLD idiom in the same pass."

---

### 7. Global rig parameter changes validated only on the reported case — breaks extreme poses
**~2 sessions** · Cost: medium (catalog split not caught until re-bake)

A global `weld k` override tuned on one figure (the swimmer, arms down) looked perfect in SSR but split a raised-arm figure in the browser bake. The constraint is a window: hanging arms need small k (de-web armpit), raised arms need large enough k (shoulder is their only torso bridge). Testing only the reported case misses the opposing constraint.

**Filed as backlog:** Would benefit from a pose-coverage test (a few extreme poses, assert single-component after weld change). Not added to CLAUDE.md — the pattern generalizes beyond weld k and needs a broader "test global rig changes against extreme catalog poses" section that warrants more thought.
