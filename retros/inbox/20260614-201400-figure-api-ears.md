## Liked / Worked
- **Prototype the geometry headlessly BEFORE touching source.** I built the
  three ear types as inline raw-SDF in a `.plans/` scratch file and rendered a
  `compare` side-by-side from a true profile angle first. That converged the
  aesthetic in two render passes, and the winning shapes ported into `buildEars`
  almost verbatim — no implement→bake→review churn on the real module.
- **Measuring the buried-eyes bug with an `evaluate()` probe beat guessing.** A
  10-line `vite-node` script walking the skin SDF along `headForward` from the
  eye anchor showed the face surface was ~1.3 units *ahead* of the anchor — so
  the fix (trim cheek/nose bulk + nudge eyes forward) was targeted, not a blind
  parameter sweep. The CLAUDE.md "measure empirically, don't trust analytic
  extents" note generalizes perfectly to "is this feature buried?".
- **Designing the hair⇄ear coupling as ONE enum on the part that owns the
  silhouette** (`hair({ ears })`) instead of a cross-part wiring kept the API
  composable and the default byte-identical (carve nothing on `'cover'`).

## Lacked
- **No pre-flight list of the figure option enums.** I burned two render cycles
  on invalid options (`mouth:'neutral'`, `sleeve:'tank'`, `boots:{height}`,
  `pants.length:'short'`) before grepping the `assertEnum` call sites. A single
  "valid figure option values" table (or a `F.optionsHelp()`) would have saved
  them — the enums are scattered across ~8 builder functions.
- **The "eyes resolve to 0 paintable triangles" trap is a recurring figure
  footgun** — it cost a full bake cycle to discover and is purely a
  consequence of `F.face.eyes` not auto-seating onto the assembled surface.
  Filed as #678. Every feature-rich face will hit it until the API seats eyes.
- **`json.dump` defaulting to `ensure_ascii=True` silently re-escaped every
  em-dash in `manifest.json`**, turning a 3-entry add into a +121/-91 churn the
  reviewer caught. Editing committed JSON via Python needs `ensure_ascii=False`
  as a reflex; re-emitting from `origin/main`'s copy collapsed it to +30.

## Longed for
- **A scriptable colored-bake check in the model:preview loop.** `model:preview`
  shades by normal, so "do the eyes/labels actually paint?" only surfaced at the
  full xvfb catalog bake (~90s each). A fast headless `paintByLabel`-resolves
  check (does each label map to >0 triangles?) would have caught the buried eyes
  in the 2s preview loop instead of the bake.
- **A `figure.optionsSchema` the way `paramsSchema` exists** — one place an
  agent can read every builder's valid option values, so option typos fail fast
  with the allowed set instead of after a render.

## Round 2 (user feedback: ear shape, hair gap, googly eyes, busts)

## Liked / Worked
- **A thin ear-shaped PLATE + shallow offset scoop is the right primitive for a
  clean ear.** My first redesign (and a delegated `model-sculpt` pass) built the
  ear from additive capsule arcs + fill blobs → a lumpy "crumpled" mass. The
  winning shape: one laterally-thin ellipsoid plate with a concha scooped from
  the OUTER face by an offset ellipsoid — leaves a natural rim + open-front bowl,
  no keyhole hole, no lump. Verified in isolation it's genuinely ear-like.
- **`smoothSubtract` fixed the hair-pocket speckle.** A *hard* sphere subtract
  for the 'behind' pocket left a sharp rim that slivered against the nearby ear
  surface (a near-tangent intersection meshes as speckle). `smoothSubtract`
  rounds the rim into a bezel that blends into the hair — clean, and it let the
  pocket sit tighter (hug the ear) instead of a wide crater.
- **Prototype-and-pick beat blind iteration for the subjective calls.** Rendering
  OLD-vs-new ear comparisons + a 3-tile pocket-tightness sweep and asking the
  user to choose (AskUserQuestion) converged "eared + tight" in one round.
- **Busts are the ideal ear showcase and bake cheaply** (head+shoulders, ~120k
  tris) — and using inline `api.paint.label` means NO palette file for the bake.
  Default-head busts also need NO eye nudge (the face doesn't bulk past the eye
  anchor), confirming the burial is specific to bulked/non-oval faces.

## Lacked
- **Thin features need their OWN detail region or they facet.** The ear plate
  came out jagged at the head grid until I added per-ear spheres to `faceDetail`
  (`earEdgeLength`). The detail-region *boundary* is also a meshing seam — make
  the fine sphere comfortably larger than any feature crossing it.
- **I over-corrected the buried-eyes fix into googly eyes.** A `headForward`
  nudge of `r.head*0.2` left the eyeball ~0.46u proud (a ball stuck on). The
  probe (round 1) gave the data to dial it to `0.12` (~0.13u cap, embedded) —
  but I should have computed the *target protrusion* the first time, not just
  "enough to paint". Tune to "just proud", not "clears the surface".
- **Catalog-only commits silently skip pr-checks.** A commit touching only
  `examples/*.js` + `public/catalog/*.json` (no `src/`/`tests/`) did NOT trigger
  the pr-checks or code-quality workflows, despite their `paths-ignore` listing
  only `*.md`. Had to validate the new entries by running `catalog.spec.ts`
  locally. If the substantive code is on an earlier commit, the HEAD commit can
  show "no required checks" — watch for it before relying on a green HEAD.

## Longed for
- **`npm run catalog:rebake -- <id...>`** (echoing the nipple retro's ask) — the
  ear-quality fixes forced re-baking all three figures by hand THREE times
  (xvfb + per-entry palette/camera flags). A one-shot rebake reading
  bake-manifest (source + gates + palette + camera) would have saved ~10 manual
  bake invocations across this session.
- **A headless "does every label resolve to >0 triangles?" check** (restated) —
  it would have caught both the buried eyes AND confirmed the new figures paint
  without the 90s xvfb bake each time.
