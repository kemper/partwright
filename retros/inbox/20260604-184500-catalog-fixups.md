Follow-up to the catalog quality pass after the human spotted three more issues.
Two more independent agents reinforced earlier themes and pinned down specifics.

## Liked / Worked
- A drab-but-functional thumbnail (the watchtower's 4-panel "box" montage) was
  fixed by simply re-baking the existing code + reconstructed `byLabel` paint —
  no model changes. Re-baking is the cheap fix when the *only* problem is that a
  saved version captured a multi-view thumbnail instead of the standard 3/4.
- For a fused replicad solid, `findFaces({box, normal})` → `paintFaces(triangleIds)`
  is the deterministic replacement for `byLabel` (which `fuseAll` scrambles).
  Confirms the robot-buddy finding: paint replicad by world coordinates, not labels.

## Lacked / Learned (reinforces the earlier "tile orientation" item — now with the value)
- **The catalog tile camera is iso azimuth 135 (the +X/−Y corner), not a flat
  axis.** A face authored on flat −Y lands off-center and foreshortens; the
  smiley-cactus face had to be `rotate([0,0,45])`'d onto the camera-facing corner.
  This is the *same* friction 6 agents hit last pass — now with the concrete
  number. The one-line doc fix is precise: "the catalog thumbnail renders at iso
  azimuth ≈135° / the +X,−Y corner — put the model's front/face there."
- `fuseAll` silently scrambles `byLabel`: the red shutter + chrome advance lever
  painted a plausible triangle count at a *hidden* location (barrel triangles), so
  the controls were invisible and never errored. Only checking the painted
  region's bbox against the feature's intended world coords catches it.

## Longed for
- Re-stated from last pass, now higher-confidence: a `--thumb-azimuth/--thumb-elevation`
  flag (or per-session thumbnail-camera pin) on the bake/export so character &
  faced models don't have to bake their orientation into geometry to face the
  iso-135 tile camera. This single tool would have removed friction in ~8 of the
  models across both passes.
